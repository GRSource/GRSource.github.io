# 架构

# 模块设计原则

# 越底层的模块，应该越稳定，越抽象，越具有高复用度

# 不要让稳定的模块依赖不稳定的模块，减少依赖

# 提升模块的复用度，自完备性有时候要优于代码复用

# 每个模块只做好一件事情，不要让Common出现

# 按照你架构的层数从上到下依赖，不要出现下层模块依赖上层模块的现象，业务模块之间也尽量不要耦合

# 解耦规则

# 公共模块下沉

# 面向接口调用

  • 模块间代码调用就会出现依赖，我们可以依赖于接口编程

# 面向协议编程

  • 对于一些统一的调用，比如页面跳转使用接口就不太方便，我们可以通过URL跳转协议来满足需求

    • 缺点

      • 对于UIImage这种对象，或者带特殊字符的URL都会发生解析错误，可以使用URL Encode，但无形中增加了调用成本

# CTMediator解耦方案

  • 基于Runtime来实现动态调用，避免模块注册，同时也避免产生反向依赖
  • 通过Category暴露出更友好的接口，避免参数的构造和传递
  • 在target-action之上构建openURL的处理

# 组件化

# 随着项目的增大，业务逻辑越来越复杂，业务之间的依赖关系更强，我们需要通过组件化让代码进行解耦。

# 组件组成部分

  • 项目主工程

  • 业务组件

  • 基础工具类组件

  • 中间件组件

  • 基础UI组件

  • 业务工具组件

    • 这类组件是为各个业务组件提供基础功能的组件。这类组件可能会依赖到其他的组件。例如：网络请求组件，图片缓存组件，jspatch组件等

# 组件化主要使用私有化CocoaPods进行划分，然后在各个子项目中实现自己的MVC、MVVM等模式

# 设计原则

# 单一职责原则

# 开闭原则

  • 业务的增加我们不应该修改原有的功能，而是通过实现其接口来增加功能

# 依赖倒置原则

  • 针对接口编程，通过继承接口或者实现接口来编程

# 接口隔离原则

# 迪米特法则

  • 一个类应该只和它的成员变量，输入输出交互，不应该引用其他类

# 里氏替换原则

  • 子类可以实现父类的抽象方法和增加自己的方法，不能重写父类的非抽象方法
